# 周辺設計書: "The Hidden Gears" (v2)

研修を「最高のもの」にするために、マニュアル（表）以外に準備すべき「裏側の設計」を定義します。

## 1. 評価設計 (Evaluation Design: The Skill Level)
研修生のレベルを可視化する。

- **Rank C (Novice)**: 言われた通りに写経できる。
- **Rank B (Practitioner)**: 致命的なバグがなく、他人が使える。
- **Rank A (Hero)**: ドキュメント（要件定義書・設計書）が充実しており、セキュリティと保守性が考慮されている。

---

## 2. 素材・テンプレート提供方針 (Material Strategy)
演習用リポジトリは配布せず、**「新人が自分でフォルダを作り、環境を構築する」** ことを重視します。

### テンプレートの提供方法
マニュアル本文（PDF）の中に「コードブロック」として記載し、それをコピー＆ペーストさせます。
「このファイルを `REQUIREMENTS.md` という名前で保存せよ」という指示自体も、ファイル操作の訓練とします。

### 画像素材
- **Day 1**: 自分の好きな画像を用意させる、またはフリー素材サイト（Unsplashなど）からダウンロードさせる。
- **Day 5**: 必要な素材（ロゴなど）がある場合は、マニュアル内のリンクからダウンロードさせる。

### テンプレートの内容 (Templates)
白紙だと書けないため、最初は「埋めるだけ」の状態から始める。

**REQUIREMENTS.md (要件定義書)**
```markdown
# 要件定義書: [プロジェクト名]

## 1. なぜ作るのか？ (背景)
- [ ] 誰が困っている？: 
- [ ] どうなれば解決？: 

## 2. 誰が使うのか？ (ターゲット)
- [ ] メインユーザー: 
- [ ] 利用シーン: 

## 3. 何を作るのか？ (機能要件)
- [ ] 画面1: [機能名]
- [ ] 画面2: [機能名]
```

**DESIGN.md (基本設計書)**
```markdown
# 基本設計書: [プロジェクト名]

## 1. 画面遷移図
(手書きメモの写真をここに貼る)

## 2. データ設計 (Data Model)
- User: { id, name, ... }
- Reservation: { id, date, ... }

## 3. ディレクトリ構成
- /components ... 部品
- /pages ... 画面
```
