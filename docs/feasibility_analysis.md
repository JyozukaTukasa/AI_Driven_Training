# 実現可能性検証 & リスク対策レポート (Feasibility Study)

ユーザーからの「本当に10日で2.5年選手のスキルになるのか？」「スケジュールに無理はないか？」という極めて重要な問いに対する分析と、それを実現するための補強策を提示します。

## 1. 「2.5年相当」の定義と実現性 (Skill Level)
現代の「AI武装済みエンジニア」における2.5年相当の定義を再設定します。

| 従来の2.5年選手 | **AI時代の2.5年選手 (本研修のゴール)** |
|:---|:---|
| 文法を暗記しており、空で書ける | 文法は忘れるが、**「何を実現したいか」を言語化できる** |
| バグの原因を直感で特定できる | エラーログを読み、**AIと一緒に仮説検証・修正ができる** |
| 時間をかけてゼロから実装する | **設計・要件定義に時間をかけ、実装は一瞬で終わらせる** |

### ギャップ分析 (Risk: Debugging Skill)
「設計(Day5)」や「セキュリティ(Day4)」はカリキュラムでカバーできますが、**「未知のエラーにハマった時の突破力（デバッグ力）」** だけは、10日間の経験量では圧倒的に足りず、ここが最大の弱点になります。

> **対策: "The Debugging Drill"の強化**
> 毎日1回、必ず**「エラーログを読む練習」**を入れます。「赤い文字が出たらAIに投げる」ではなく、「赤い文字の『どの部分』が重要かを特定してからAIに投げる」訓練を徹底します。

## 2. スケジュールの現実性 (Schedule Risk)
特に **Day 5 (要件定義～実装～発表)** の密度が危険水域です。未経験者が1日でこれを行うのは通常不可能です。

### Day 5 のタイムテーブル分析
- 10:00 - 12:00: 要件定義・設計 (Mission 1-2) ← ここが重い
- 13:00 - 16:00: 実装 (Mission 3) ← 未経験者はここでハマる
- 16:00 - 17:00: 修正・発表準備
- 17:00 - 18:00: 成果発表

### 対策: "Asset Reuse" & "Golden Path"
Day 5 を完走させるために、**「機能削減」ではなく「効率化」** で勝負させます。

1.  **Asset Reuse (資産の再利用)**:
    -   **Day 4のコード（認証・DB接続）をコピー＆ペーストして流用する** ことを公式に推奨する。
    -   「一度作ったものを二度書かない」のがプロの流儀であると教える。
    -   これにより、ログイン実装にかかる時間を「1時間」から「5分」に短縮する。
2.  **Golden Path (正解ルートの用意)**:
    -   どうしても動かない場合のために、マニュアルの巻末（または別冊）に **「正解コード一覧（虎の巻）」** を用意する。
    -   どうしても15:00時点で動かない場合、**「ここからコピペして良い」** ルールにする。
    -   「納期（17:00）を守るために、プライドを捨ててコードを借りる」のも重要なプロの判断（Trade-off）と教える。

---

## 3. 結論 (Conclusion)
- **Q**: 2.5年相当になるか？
    - **A**: **「設計力・提案力」はなります。** 「自力解決力」はまだ足りませんが、「AIと共に解決する力」でカバー可能です。
- **Q**: スケジュール通りできるか？
    - **A**: Day 3(API) と Day 5(App) が鬼門です。**「スコープ調整（機能削減）」** の判断をMentor Botにさせることで、時間内の完走を保証します。

この「リスク管理」も含めてマニュアルに記載し、**「間に合わない時は機能を削る決断をせよ」** と教えること自体を教育の一部とします。
